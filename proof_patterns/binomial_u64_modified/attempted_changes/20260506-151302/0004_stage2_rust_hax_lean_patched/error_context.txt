After replacing trailing_zeros with manual shift loops and adding loop_decreases! to all three Stein's loops, lake build now fails twice on `hax_construct_pure: mvcgen generated more than one goal containing the metavariable. ... Try to remove if-then-else and match-constructs.` (binomial_u64.lean:38:7 and 72:7). The if-then-else inside `loop_decreases!(if m < n { n } else { m })` cannot be reduced to a pure value by Hax's tactic. Stein's algorithm has no single-expression measure without if-then-else (max(m,n) is essential, m+n could overflow without a precondition). Switching to Euclidean GCD removes the issue: the measure `b` strictly decreases with no if-then-else and no overflow risk.
