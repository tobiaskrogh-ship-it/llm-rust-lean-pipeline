After diagnostic narrowing of the lake build error to a `pureTermination`/`hax_construct_pure` synthesis failure, the root cause is that Hax's `hax_construct_pure` tactic (used by `rust_primitives.hax.while_loop`) cannot handle `loop_decreases!` measures that are anything but a single variable read — its own error message says "mvcgen generated more than one goal containing the metavariable. This is currently unsupported. Try to remove if-then-else and match-constructs." The previous gcd measure `if m > n { m } else { n }` produced a `sorry` warning + Lean exit code 1; `m | n` produced a `failed to synthesize` error. Refactor: switch gcd to Euclidean (single-variable `b` measure, no `trailing_zeros` calls); add explicit countdown variables `steps` (in `binomial`) and `remaining` (in `multinomial`) so that every `loop_decreases!` measure is a single variable read. Drop the now-unused `trailing_zeros_u64` helper.
