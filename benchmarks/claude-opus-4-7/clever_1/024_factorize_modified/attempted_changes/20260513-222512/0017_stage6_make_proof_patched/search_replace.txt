### Why this is left as `sorry`

This single lemma combines four proof patterns the reference library
does not currently expose **together**:

* `partial_fixpoint` strong-induction with a non-obvious two-parameter
  measure (`n` decreases on divide, `p` increases on trial ‚ÄĒ c.f.
  `gcd_recursive_modified` for the one-parameter case and
  `000_has_close_elements_modified` for `n*n - k`).
* Vec-output construction across recursive calls (c.f.
  `rescale_to_unit_modified` / `clever_009_rolling_max_modified`,
  which are well-founded, not `partial_fixpoint`).
* Product-of-Vec invariant maintained across pushes ‚ÄĒ no reference
  example covers this.
* Per-element primality + sortedness invariants on a built Vec ‚ÄĒ no
  reference example covers this.

**Structural unblock**: a separately verified
`partial_fixpoint`-with-Vec-output proof pattern in the library
(combining `gcd_recursive_modified`'s `Nat.strongRecOn` style with
`rolling_max`'s `extend_from_slice` step lemma), plus a small number
of arithmetic lemmas about trial-division (e.g.
`IsPrime_of_smallest_divisor_at_least_p`, `IsPrime_of_p_sq_gt_n`)
would each close one component of the bundle. The Vec-output side
maps directly onto the existing `push_one` helper pattern. -/

### Progress in this pass (attempt 2 of 2)

* The strong-induction skeleton is set up via `suffices aux` over the
  combined measure `n.toInt.toNat * 2^32 + (2^32 - p.toInt.toNat)`,
  which decreases strictly on both branch (b) (by ‚Č• 2^32, since
  `n` shrinks) and branch (c) (by exactly 1, since the `(2^32 - p)`
  term decreases). This unblocks the "missing two-parameter measure"
  issue called out in the previous attempt.
* The base case (`n ‚Č§ 1`) is fully closed.
* The outer-if reduction in the recursive case is performed
  (`decide (n ‚Č§ 1) = false`).
* The supporting helper `isPrime_of_frontier_and_pp_gt_n` is fully
  proved ‚ÄĒ `n` is prime when no integer in `[2, p)` divides it and
  `p¬≤ > n`. The classical small-divisor / complementary-divisor
  argument is discharged using `Int.mul_le_mul_of_nonneg_*` and
  `Int.mul_neg_of_pos_of_neg`. This kills the
  "no per-element primality reference example" gap.

### What remains stuck

The three inner branches `(p *? p > n)`, `(n %? p == 0)`, and the
fall-through. Each needs the same four ingredients (overflow-free
discharge of the partial op, decisional reduction, Vec push-step,
and ‚ÄĒ for the recursive branches ‚ÄĒ IH application with re-established
preconditions and a measure-decrease witness).

**Stuck sub-goal**: after reducing the outer `if`, the goal is

  `‚ąÉ v, factorize_at <unfolded inner body> n p acc = RustM.ok v ‚ąß ...`

where `<unfolded inner body>` is the three-way `if`-chain. Discharging
the `p *? p` reduction is the hardest single step ‚ÄĒ it needs a
BitVec `smulOverflow = false` proof from `p ‚Č§ n ‚Č§ 2^31 - 1`, mirroring
the `Int64.subOverflow` discharge in `largest_divisor_modified` but
for signed multiplication. The other reductions (`n %? p`, `n /? p`,
`p +? 1`) follow the same pattern as `largest_divisor_modified`'s
`h_rem` / `h_sub` blocks.

### Structural unblock

A single combined reference example covering `partial_fixpoint` WITH
Vec-output construction (the gap explicitly called out by the picker:
"No example combines `partial_fixpoint` *with* Vec-output construction")
would let the next pass copy the per-step structure mechanically. The
arithmetic helpers needed (primality lemma; `p¬≤ ‚Č§ 2^62` bound from
`p ‚Č§ 2^31`) are local and small. -/
