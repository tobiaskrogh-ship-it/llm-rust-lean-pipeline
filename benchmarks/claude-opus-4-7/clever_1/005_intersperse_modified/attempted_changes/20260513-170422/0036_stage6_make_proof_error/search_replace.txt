/-- Helper for totality: under the invariant
    `acc.val.size = if i.toNat = 0 then 0 else 2 * i.toNat - 1`,
    `intersperse_at` returns `ok v` for some `v`.

    **Stuck sub-goal.** In the `i > 0` branch of the strong induction we
    need to apply `intersperse_at_step_pos`, whose `h_no` precondition is
    `acc.val.size + 2 < 2 ^ 64`. Substituting the invariant
    `acc.val.size = 2 * i.toNat - 1` reduces this to
    `2 * i.toNat + 1 < 2 ^ 64`. From the available hypotheses
    (`i.toNat < numbers.val.size`, `numbers.val.size < 2 ^ 64`) this is
    *not* derivable in the regime `2 ^ 63 ≤ numbers.val.size < 2 ^ 64` —
    pick `i.toNat = 2 ^ 63 - 1` to falsify `2 * i.toNat + 1 < 2 ^ 64`.

    **Structural unblock.** The realistic external change is to add the
    precondition `h_size_bound : 2 * numbers.val.size ≤ 2 ^ 64` to
    `intersperse_total` (propagated through `intersperse_at_total_aux`).
    Under that precondition, the stuck sub-goal closes by `omega`:
    `2 * i.toNat + 1 ≤ 2 * (numbers.val.size - 1) + 1
       = 2 * numbers.val.size - 1 < 2 ^ 64`. The Rust proptest in
    `src/lib.rs` restricts lengths to `0..32`, so the precondition holds
    in every realistic execution; the obligation as currently stated is
    mathematically false for `numbers.val.size ≥ 2 ^ 63`. An alternative
    unblock is a separately-verified Hax prelude lemma tightening the
    `RustSlice` invariant to `2 * val.size ≤ USize64.size`, which would
    eliminate the need for the explicit precondition. -/
